## Exit criteria

> At least 80% of surfaced roles are plausibly relevant, duplicates stay
> below 5%, and broken sources fail visibly rather than silently.
>
> Relevance is now measured, not assumed: `AnalyticsSection`'s new "Source
> coverage" panel shows exactly this — total surfaced, excluded count, and
> relevance rate per source, flagged red below 80%. Duplicate rate isn't a
> separate metric — the unique `(user_id, fingerprint)` constraint makes a
> true duplicate structurally impossible to insert twice, so "below 5%"
> holds by construction rather than needing to be watched. Broken sources
> fail visibly via `source_runs` (status, error, surfaced in the Sources UI)
> rather than silently. What's outstanding: this has all been verified with
> hand-seeded and one real live run, not weeks of real accumulated
> discovery data — the exit criteria are architecturally satisfied but not
> yet empirically proven over time, which is exactly the gap task 5.9 asks
> to measure before spending money on a paid API.

## Notes

- **`JobSourceAdapter` (`src/lib/job-sources/types.ts`) has three methods**: `discover(config)` hits the source and returns raw postings with a stable `externalId`; `normalize(raw, config)` reshapes one raw posting into GYST's `NormalizedJob`; `healthCheck(config)` does a cheap reachability check independent of a full discovery run, so a broken source can be flagged without pulling all its postings. `config` is the `source_configs.config` jsonb — adapter-specific, so adding an adapter never needs a schema migration.
- **Role-family/SWE/finance classification is keyword-based, in `src/lib/job-sources/classify.ts`, not AI** — CLAUDE.md keeps deterministic logic out of prompts, and there's no AI provider wired up yet anyway (`src/ai/index.ts` still returns `null`). It's a first triage pass only; once an opportunity is saved, the existing Phase 4 score-edit UI still lets the user correct it.
- **Two real, unauthenticated ATS adapters**: Greenhouse (`boards-api.greenhouse.io`) and Lever (`api.lever.co`) — both public read APIs with no ToS restriction on this kind of access, matching PLAN.md §8 source order #1. `discover()` filters to internship titles before normalizing (`isInternshipTitle` in `classify.ts`, word-boundary-aware — a naive `/intern/i` also matches "International"/"Internal", caught during live verification against Airbnb's real board and locked in with a fixture case), since an unfiltered company board is mostly full-time roles irrelevant to a Summer 2027 internship search.
- **Curated feed adapter (task 5.3) reads Pitt CSC/Simplify's `Summer2026-Internships` `listings.json`** — a public, unauthenticated file the repo generates specifically for programmatic consumption (other tools like swelist.com already consume it the same way). Terms review: no auth wall, no robots restriction on raw.githubusercontent.com, and the repo's own README advertises third-party tools built on this feed. The repo gets renamed each cycle (there's no `Summer2027-Internships` repo yet as of 2026-07-12, though the current repo already carries some `"Summer 2027"`-tagged listings) — `feedUrl` lives in `source_configs.config` so switching repos next cycle is a config edit, not a code change. Default filters: category `"Product"`, term `"Summer 2027"`.
- **Discovery orchestration lives in `src/lib/job-sources/{ingest,run-discovery}.ts`.** `ingestNormalizedJob` upserts by the existing `opportunityFingerprint` (so the same role surfacing from two sources collapses into one row), writes new discoveries to the `discovered` application stage (distinct from Phase 4's manual-capture `saved`, so they queue for triage instead of looking already-decided), and reuses `scoreOpportunity`/`buildJobScoreRow` — the exact same hard-exclusion and scoring path the manual capture action uses. `runDiscoveryForSource` records every run in `source_runs` (success or error) and expires (`active = false`) any previously-active opportunity from that source no longer present in the current run.
- **`findOrCreateCompany` moved from `src/app/(app)/recruiting/company-helpers.ts` to `src/lib/companies.ts`** so both the manual capture action and the discovery pipeline can call it; same for a new `getTargetGradYear` helper (`src/lib/recruiting-preferences.ts`) and a shared `buildJobScoreRow` (`src/lib/job-scoring.ts`) — no behavior change, just de-duplicating what would otherwise be copy-pasted between the two ingestion paths.
- **Scheduling: Vercel Cron, not Supabase Edge Functions.** PLAN.md §4 named either as an option; Vercel Cron hitting a normal Next.js route handler (`/api/cron/discover-jobs`, `vercel.json`) reuses the same server code/deploy pipeline instead of a second runtime, which fits "low maintenance" better for a single-user app. The route checks `Authorization: Bearer $CRON_SECRET` (Vercel sends this automatically for scheduled invocations once `CRON_SECRET` is set) and runs via a new service-role client (`src/lib/supabase/service.ts`) since a scheduled job has no user session — the one addition to `src/lib/env.ts`/`.env.example` this task needed.
- **The auth proxy (`src/lib/supabase/middleware.ts`, formerly `middleware.ts` — Next 16 renamed the convention) was redirecting `/api/cron/*` to `/login`** before the route handler ever ran, since it treated "no Supabase session" as "redirect," with no exception for routes that authenticate a different way. Fixed with a `BEARER_AUTH_PATHS` bypass list. Caught during live verification, not by the unit suite — worth calling out since it's exactly the kind of bug that only shows up hitting the real route.
- **Discovered opportunities don't mix into the main pipeline.** `recruiting/page.tsx` splits one `applications` query into `pipelineApplications` (`stage !== "discovered"`, fed to the existing board/table/follow-ups, unchanged) and `discoveredApplications` (fed to the new `DiscoveryQueue`) — PLAN.md's anxiety-aware UX principle rules out dumping an automated firehose next to roles the user already decided to pursue. `DiscoveryQueue` sorts by the same `sortByScore` the table already used, and hard exclusions (already computed by `scoreOpportunity` during ingestion) show as the same destructive "excluded" badge as everywhere else — no new ranking logic needed, task 5.5 was mostly a surfacing problem.
- **Feedback (5.6) reuses the score-edit action's math, not a new scoring path.** `setOpportunityFeedback` (`src/app/(app)/recruiting/feedback-actions.ts`) sets `user_feedback_score` to 5 ("up") or 0 ("down") and recomputes `total_score` the same way `updateJobScore` does, then stores the raw `up`/`down`/`not_relevant` choice on `opportunities.feedback` (new column from task 5.1's migration, `check` constraint enforced — verified live that an invalid value is rejected). "Not relevant" additionally archives the application via the same from/to `application_events` pattern every other stage transition uses, so it drops out of the queue for good — re-ingestion only ever touches `last_seen_at`/`active`, never `stage`/`feedback`.
- **"Save" promotes `discovered` → `saved` through the existing `updateApplicationStage` action** — no new promotion code path, since that action already does the right thing (stage update + `application_events` row + revalidate).
- **Bookmarklet, not a browser extension (task 5.7).** LinkedIn/Handshake have no viable personal-use API (PLAN.md §8), and a Manifest V3 extension needs store review plus per-browser maintenance for a feature this small — the bookmarklet PLAN.md offers as the lighter alternative just opens `/recruiting/capture?url=...&title=...` with the current page's `location.href`/`document.title` (`src/components/recruiting/bookmarklet-card.tsx`). The capture page (`recruiting/capture/page.tsx`) lives inside the same `(app)` auth boundary as everywhere else — no new auth surface — and just pre-fills the existing `OpportunityForm` (now takes optional `initialValues`/`defaultOpen` props) rather than duplicating the form.
- **Closing soon (`closing-soon.tsx`) mirrors `follow-ups-due.tsx`'s existing pattern** — same terminal-stage exclusion set plus `discovered` (untriaged roles don't belong here either), 14-day window, sorted soonest-first, renders nothing rather than an empty card when there's nothing to show. Caught a `react-hooks/purity` lint failure from using `Date.now()` directly in a Server Component render — swapped for `new Date().getTime()` to match the existing convention (`follow-ups-due.tsx` already did it that way; the rule flags the bare `Date.now` call specifically, not `new Date()`).
- **Weekly digest (`src/lib/job-sources/weekly-digest.ts`) counts the same three things the page already surfaces** (discovery queue size, closing-soon count, follow-ups due) via count-only Supabase queries (including one PostgREST embedded-resource filter — `opportunities!inner(...)` joined so `closingSoon` can filter on the opportunity's `deadline`/`active` from the `applications` table) and sends **one** push notification only if at least one count is non-zero — an empty week gets silence, not a "you have 0 things to do" ping, per PLAN.md's no-guilt-copy/no-noise UX principle. Reuses `createNotification`/`sendPush` from Phase 3 unchanged; `NotificationKind` gained a `"digest"` value (migration widens the `notifications_kind_check` constraint to match). Scheduled Monday 14:00 UTC in `vercel.json`, same bearer-secret cron auth as discovery.
- **Source coverage (task 5.9) is a measurement, not the paid-API integration itself** — PLAN.md explicitly sequences "evaluate a paid search API only after measuring coverage gaps," so this task adds `computeSourceCoverage` (`src/lib/recruiting-analytics.ts`, groups by `opportunities.source`, relevance rate = `1 - excluded/total` using the same hard-exclusion flag `scoreOpportunity` already computes) and a matching panel in `AnalyticsSection`, and stops there. **No paid API was evaluated or integrated.** Revisit only once real discovery has run for a few weeks and this panel actually shows a source with a real coverage gap (relevance rate persistently low, or free sources combined not surfacing enough volume) — evaluating a monthly-cost API against a hypothetical gap would violate PLAN.md §18's cost discipline for a single-user app.
- **Verification**: unit tests (26 for the adapters/classification/ingest/run-discovery pipeline, fixture-based, network-independent) plus two live round trips against the real Supabase project (service-role client, insert → exercise → read back → cleanup, the same pattern Phase 4 used since this sandbox can't complete a browser magic-link sign-in): (1) through the running dev server — inserted a real `source_configs` row pointing at Airbnb's live Greenhouse board, hit `/api/cron/discover-jobs` for real, confirmed one real opportunity ("Sales Operations Intern, Italy") landed with a correct `job_scores` row and a `discovered`-stage application, re-ran it to confirm idempotency (0 created, 1 updated, 0 duplicates), confirmed the bearer-auth guard 401s on missing/wrong secrets while `/recruiting` is still session-redirected; (2) the feedback/promote math directly against the schema — "up" recomputed `total_score` correctly (36→38), "not relevant" archived the application with a correct `application_events` row, and the `opportunities_feedback_check` constraint rejected an invalid value. Everything was deleted afterward. Confirmed `/recruiting/capture` redirects to `/login` when unauthenticated, same as every other app page, and an RTL test covers `OpportunityForm` rendering pre-filled from `initialValues`. The weekly digest was hit live twice through the dev server — once with an empty account (`sent: false`, all counts 0, no notification row) and once after seeding one `discovered` opportunity (`sent: true`, correct count, and a real `notifications` row with the right `body` text) — then cleaned up. The coverage query's PostgREST shape (`applications` selecting a doubly-nested `opportunity:opportunities(source, role_family, job_scores(excluded))`) was also confirmed live by seeding one excluded opportunity and reading it back before cleanup. What's *not* verified: a real click-through of the Discovery queue / bookmarklet-to-capture / analytics UI in an authenticated browser session, same caveat Phase 4 flagged for its own UI. `npx tsc --noEmit`, `eslint`, and `next build` are all clean; the suite is 153 tests across 35 files.
